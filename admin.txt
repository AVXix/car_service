Admin module reference (`admin.php`)
====================================

1) File structure (admin side)
------------------------------
- `admin.php`
  - Loads `config.php`, starts session, runs `handlers/admin_page_request_handler.php`.
  - Chooses one view:
    - `views/admin_authentication_section.php` when admin is not logged in.
    - `views/admin_dashboard_section.php` when admin is logged in.
  - Loads `js/admin.js`.

- `handlers/admin_page_request_handler.php`
  - Admin request orchestrator.
  - Prepares shared variables used by admin views.
  - Loads:
    - `handlers/admin_auth_request_handler.php`
    - `handlers/admin_dashboard_request_handler.php` (only for authenticated admin).

- `handlers/admin_auth_request_handler.php`
  - Handles admin auth actions:
    - `login`
    - `logout`
  - Updates admin session state and redirects back to `admin.php`.

- `handlers/admin_dashboard_request_handler.php`
  - Handles dashboard actions:
    - `update_total_slots`
    - `update_appointment`
    - `cancel_appointment`
  - Loads dashboard data:
    - mechanics list
    - appointments list
    - optional edit appointment data from `?edit={id}`.

- `views/admin_authentication_section.php`
  - Renders admin login form and login error.

- `views/admin_dashboard_section.php`
  - Renders:
    - top logout action,
    - mechanics table with inline slot update form,
    - appointment edit form (when edit mode is active),
    - appointment list with edit/cancel actions.

- `js/admin.js`
  - Frontend behavior:
    - auto-submit mechanic slot form when value changes,
    - prevent negative slot input,
    - confirm logout and edit submit actions.


2) How implementation works (admin request flow)
------------------------------------------------
Step 1: Entry
- Browser opens `admin.php`.
- Common setup runs (`config.php`, session start, admin page handler).

Step 2: Handler orchestration
- `admin_page_request_handler.php` initializes all admin view variables.
- Auth handler always runs first.
- If `$_SESSION['admin_id']` exists, dashboard handler runs.

Step 3: Action processing
- Auth handler processes login/logout POST actions.
- Dashboard handler processes slot updates, appointment updates, and appointment cancellation.

Step 4: Data loading
- Dashboard handler fetches mechanics + slot stats.
- Dashboard handler fetches appointment table data.
- If `?edit={id}` is present, it loads one row to prefill edit form.

Step 5: Rendering
- Not authenticated -> login view.
- Authenticated -> dashboard view with tables/forms.
