Index module reference (`index.php`)
====================================

1) File structure and dependencies
----------------------------------
- `index.php`
  - Uses: `config.php`, `handlers/index_page_request_handler.php`, `views/authentication_section.php`, `views/appointment_booking_section.php`, `js/index-page-interactions.js`
  - Role: entry/composition page only.

- `handlers/index_page_request_handler.php`
  - Uses: `handlers/index_auth_request_handler.php`, `handlers/index_booking_request_handler.php`
  - Role: shared state bootstrap (errors/messages/view mode/form defaults), CSRF helpers, flash-message helpers.

- `handlers/index_auth_request_handler.php`
  - Uses: `$conn` from `config.php`, helper functions from `index_page_request_handler.php`
  - Role: handles `create_account`, `sign_in`, `logout_user`.

- `handlers/index_booking_request_handler.php`
  - Uses: `$conn`, booking/shared state from `index_page_request_handler.php`
  - Role: loads mechanics, validates booking, enforces duplicate/capacity rules, inserts appointment.

- `views/authentication_section.php`
  - Uses: `$authMode`, `$authErrors`, `$authNotification`, `$csrfToken`
  - Role: renders sign-in or create-account forms.

- `views/appointment_booking_section.php`
  - Uses: `$mechanics`, `$errors`, `$notification`, `$csrfToken`, `$_SESSION['username']`
  - Role: renders booking form + sign-out form.

- `js/index-page-interactions.js`
  - Role: UX-only helpers (autofocus, min date, sign-out confirmation, double-submit prevention).


2) Workflow (request lifecycle)
--------------------------------
Step A: `index.php` boot
- Load DB config.
- Start session.
- Run `handlers/index_page_request_handler.php`.

Step B: request bootstrap
- Initialize shared variables for views.
- Ensure CSRF token exists in session.
- Load flash message from session (if any).
- Dispatch to auth handler and booking handler.

Step C: auth actions
- `create_account`: validate inputs -> check unique username -> hash password -> insert -> set session -> redirect.
- `sign_in`: validate inputs -> load user -> verify hash -> regenerate session id -> set session -> redirect.
- `logout_user`: verify CSRF -> clear session user keys -> redirect.

Step D: booking action
- Only runs for logged-in users.
- Validate CSRF + required fields + valid date.
- Check duplicate same-day booking by phone.
- Check mechanic capacity for selected date.
- Insert appointment and redirect with success flash.

Step E: view render
- If not logged in (`empty($_SESSION['user_id'])`) -> render `views/authentication_section.php`.
- If logged in -> render `views/appointment_booking_section.php`.


3) Most important concepts used
--------------------------------
- Session-based auth
  - User login state is stored in `$_SESSION['user_id']` and `$_SESSION['username']`.
  - Views use this state to decide whether to show auth forms or booking UI.
  - Auth handlers set/clear these values; they are the single source of login truth.

- CSRF protection
  - Every POST form includes hidden `csrf_token`.
  - Token is generated once per session and kept in `$_SESSION['csrf_token']`.
  - Handler compares posted token with session token using `hash_equals`.
  - Goal: block forged cross-site POST requests.

- Flash messages
  - One-time messages are stored in session (`index_page_flash`).
  - After redirect, bootstrap maps flash into `$authNotification` or `$notification`.
  - Message is deleted immediately after read, so it appears only once.

- PRG pattern (Post/Redirect/Get)
  - After successful POST actions, code redirects to the same page.
  - Browser refresh then repeats a GET (not the original POST).
  - Prevents accidental duplicate account creation/booking inserts.

- Prepared statements
  - All SQL operations with user input use `prepare` + `bind_param`.
  - Parameters are sent separately from SQL text.
  - This protects against SQL injection and keeps types explicit.

- Password hashing
  - New passwords are never stored as plain text.
  - Signup uses `password_hash(...)` to store a hash.
  - Login uses `password_verify(...)` to compare plain password with stored hash.

- Date-only booking model
  - Booking input and storage use `Y-m-d` only.
  - Duplicate and capacity checks compare by date (not time-of-day).
  - This matches business behavior where only day-level scheduling is required.
